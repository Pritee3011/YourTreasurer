Project Description:YourTreasurer is a comprehensive, full-stack personal finance application designed specifically for the campus ecosystem. It moves beyond simple expense tracking to act as a digital CFO for students, managing monthly allowances, social liabilities (friend loans), and recurring campus costs (EMIs, rent, and mess fees).
Built with a vintage-modern aesthetic inspired by the "Odyssey" theme, the app balances high-end UI animations with rigorous backend automation. It leverages a modern tech stack to provide real-time financial oversight, automated debt reminders, and proactive budget guarding.
Core Objectives
Financial Literacy: To help students visualize their spending habits across categories like Junk Food, Educational, and Travel.
Liability Management: To automate the awkward process of tracking and reminding friends about borrowed money.
Temporal Control: To implement a strict 30-day budget cycle that resets and archives data automatically.
Proactive Guarding: To provide tiered email alerts (10%, 5%, and 0% remaining) before a student overspends.

This is the complete YourTreasurer Implementation Blueprint. This roadmap is designed to guide participants through a professional full-stack development cycle, moving from basic database architecture to advanced automated communication and high-end UI/UX.
Competition Standard Rules
 1. The "Zero-Persistence" Rule
The application must not store user states in local variables. To ensure real-time accuracy, Name and Password must be verified against MongoDB Atlas for every major data fetch. If the credentials do not match, the app must redirect to the Budget Gateway.
2. Automatic Collection Creation
Participants are forbidden from manually creating collections in the MongoDB Atlas UI. The code must be written such that collections (users, daily_expenses, friend_loans, etc.) are automatically generated by Atlas upon the first insert_one or update_one command.
3. The 30-Day Temporal Reset
The backend must track the start_date of each budget. On every login, the logic must compare the current system date with the start date. If 30 days have passed, the app must:
Archive current expenses.
Reset total_spent to 0.
Prompt the user to set a new monthly_limit.
4. Absolute Data Integrity
Hard-coding data for the Analysis or History pages is strictly prohibited.
Verification: Judges will ask to see your MongoDB Atlas "Browse Collections" tab.
Match Check: The values on your screen must match the documents in Atlas exactly.
5. The "Guardian" Mail Logic
Automated communication is mandatory. The system must trigger Flask-Mail notifications at specific thresholds:
10% Remaining: Alert for caution.
5% Remaining: Critical warning.
0% / Over-Budget: A "Stop" alert sent for every new expense added while in a deficit.
 6. The Friend-Loan Handshake
When a loan is logged, the app must send an immediate notification email to the friend's email address. This creates a digital record. Additionally, the "Remind Friend" button must trigger a follow-up email with a "Whoosh" sound effect.
 7. Real-Time Progressive UX
The Progress Bar on the Home page and the Doughnut Chart on the Analysis page must be dynamic.
AJAX/Fetch: Adding an expense should ideally update the progress bar without a full page refresh.
Toggles: The Analysis graph must support a toggle between Day View and Month View using Chart.js.
 8. Asset Optimization
Since the app uses high-end SVG backgrounds and shimmering animations:
Cloudinary: All receipt images must be hosted on Cloudinary; storing images directly in the static folder is prohibited.
SVG Polish: The background must maintain its shimmering light effect across all sub-pages.
 9. Multi-Sensory Feedback
Implementation of specific sound effects is required at these triggers:
Track Now: "Coin Clink" sound.
Delete Expense: "Paper Crumple" sound.
Message Sent: "Whoosh" sound.
Loan Returned: "Ka-ching" sound.
 10. Connection Autonomy
Participants MUST use their own MongoDB Atlas cluster and Cloudinary credentials.
Environment Variables: Connection strings and API keys must be stored in a .env file or as environment variables on the deployment platform (e.g., Render). Do not hardcode secrets..

Phase 1: Foundation & Temporal Control
Task 1 [40 Marks]: The Secure Budget Gateway (Low Difficulty)
Files: profile.html, app.py
Database: users collection.
Detailed Logic: 1. Create a login form. If the name is found, verify the password.
2. 30-Day Logic: Fetch start_date. If datetime.now() > start_date + 30 days, wipe the current_spend to 0 and update start_date to today.
3. If the user is new, save their Name, Password, and monthly_limit.
Effect: A "Digital Unlock" chime sound on successful login.
Task 2 [60 Marks]: Daily Expense & Loan Entry (Medium Difficulty)
Files: index.html, app.py
Database: daily_expenses collection.
Detailed Logic: 1. A form with Category (Dropdown), Amount, and a "Is it a Loan?" checkbox.
2. If it's a loan, reveal extra fields: Friend's Email and Relationship.
Effect: A "Money Drop" (falling coins) animation on the screen and a "Coin Clink" sound on success.
Task 3 [60 Marks]: Real-Time Spend History (Medium Difficulty)
Files: expenses.html, app.py
Detailed Logic: 1. Perform a db.daily_expenses.find({"username": session_user}).
2. Minimum 8 dummy expenses across categories must be inserted to prove functionality.
Effect: CSS Slide-in-from-bottom animation for each expense card.
Task 4 [40 Marks]: Expense Deletion (Low Difficulty)
Files: expenses.html, app.py
Detailed Logic: Use ObjectId from bson to delete a specific record via a delete button on the card.
Effect: A "Paper Crumple" sound effect when the record is removed.

Phase 2: External APIs & Automation
Task 5 [80 Marks]: Cloudinary Receipt Integration (Hard Difficulty)
Files: app.py, index.html
Detailed Logic: 1. Integrate the Cloudinary SDK.
2. When logging an expense, upload the image file and save the secure_url in the database.
Effect: A Loading Spinner on the "Track Now" button during the upload.
Task 6 [100 Marks]: The "Guardian" Mail Tiers (Hard Difficulty)
Files: app.py (Flask-Mail)
Detailed Logic: 1. Trigger specific emails based on budget usage: 10% left, 5% left, and Over-budget.
2. Use a flag in the DB to ensure each alert (except over-budget) only sends once per month.
Effect: The Home page title Pulses Red if the user is in the "Over-Budget" state.
Task 7 [80 Marks]: Automated Loan Handshake (Hard Difficulty)
Files: app.py
Detailed Logic: If "Is it a Loan" is checked in Task 2, Flask-Mail must immediately send a notification to the Friend’s Email regarding the debt.
Effect: Display a "Lent to [Friend Name]" badge on the history card.

Phase 3: Visual Intelligence
Task 8 [60 Marks]: Aggregation API (Medium Difficulty)
Files: app.py
Detailed Logic: Create a route /api/stats that uses MongoDB $group to return the total sum per category in JSON format.
Task 9 [100 Marks]: Real-Time Doughnut Dashboard (Hard Difficulty)
Files: analysis.html, Chart.js
Detailed Logic: Fetch the JSON from Task 8 and render it in a Doughnut Chart.
Effect: Chart slices must Animate/Rotate into place on page load.
Task 10 [100 Marks]: Day vs. Month Toggle Graph (Hard Difficulty)
Files: analysis.html, Chart.js
Detailed Logic: 1. Add a toggle button.
2. Day view: Compare hourly spending of today.
3. Month view: Compare daily spending of the last 30 days.
Effect: Use chart.update() for a smooth transition between views without page reload.

Phase 4: Liabilities & Recurring Spend
Task 11 [60 Marks]: Interval Spend Manager (Medium Difficulty)
Files: interval_spend.html, app.py
Database: recurring_payments collection.
Detailed Logic: Log EMIs/Rent. Form must ask for: Amount, Due Date, and "Remind me X days before."
Task 12 [80 Marks]: Smart Reminder Automation (Hard Difficulty)
Files: app.py
Detailed Logic: Check current_date == Due Date - X. If true, send a mail to the User.
Effect: Any bill due within 3 days must have a Glowing Red Pulsing Border.
Task 13 [80 Marks]: Debt Recovery Logic (Hard Difficulty)
Files: expenses.html, app.py
Detailed Logic: A "Remind Friend" button on loan cards. When clicked, it auto-sends a "Gentle Reminder" mail to the friend's email.
Effect: Play a "Message Sent/Whoosh" sound on success.

Phase 5: The "Treasurer" Finish
Task 14 [100 Marks]: Real-Time Progress Bar (Hard Difficulty)
Files: index.html
Detailed Logic: Calculate (Total Spent / Budget) * 100.
Fetch data from DB.
If user adds an expense, the bar must update immediately (via AJAX or page refresh).
Effect: The bar color changes from Green → Yellow → Red as it fills up.
Task 15 [100 Marks]: The "Money Return" Celebration (Hard Difficulty)
Files: expenses.html, app.py
Detailed Logic: A "Mark as Returned" button for loans.
Update loan status in DB.
Add a negative entry in daily_expenses so the Progress Bar (Task 14) moves backward.
Effect: Trigger a massive Confetti Explosion and a "Cash Register/Ka-ching" sound.
